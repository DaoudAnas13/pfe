\chapter{Sprint 1: Foundation and Core Infrastructure}

\section{Introduction}

Sprint 1 established the technical foundation of RM and created the minimum viable platform infrastructure required for subsequent business features. This sprint focused on identity and access management, project setup, baseline database design, and core backend scaffolding.

The objective was to secure the platform early, define stable architectural conventions, and deliver a deployable base increment that could host intervention management features in Sprint 2.

\section{Preview of Sprint}

\subsection{Sprint Objectives}

\begin{enumerate}
\item Initialize the backend, web, and mobile workspaces with coherent configuration.
\item Implement authentication and authorization with role-based access control.
\item Create initial database schema for users, roles, and core audit entities.
\item Deliver first API endpoints and shared error-handling conventions.
\item Validate end-to-end login flow from web and mobile clients.
\end{enumerate}

\subsection{Sprint Scope Statement}

This sprint does not target complete business coverage. It targets infrastructural readiness and controlled access so that feature sprints can be implemented on a secure and consistent foundation.

\section{Functional Specifications}

\subsection{Detailed Use Cases}

\textbf{UC-S1-01: User Authentication}

\textbf{Primary Actor:} Platform User (Technician, Supervisor, Coordinator, Admin)

\textbf{Preconditions:}
\begin{itemize}
\item User account exists and is active.
\item Authentication service is reachable.
\end{itemize}

\textbf{Main Flow:}
\begin{enumerate}
\item User submits email and password.
\item Backend validates credentials.
\item Backend issues JWT token with role claims and expiration.
\item Client stores token securely and starts authenticated session.
\item User is redirected to role-appropriate dashboard.
\end{enumerate}

\textbf{Postconditions:}
\begin{itemize}
\item Authenticated session is active.
\item Access to protected resources is enabled according to role policy.
\end{itemize}

\textbf{UC-S1-02: Role-Based Endpoint Authorization}

\textbf{Primary Actor:} System

\textbf{Main Flow:}
\begin{enumerate}
\item Client requests protected endpoint.
\item Security filter validates JWT signature and expiration.
\item Authorization layer resolves role permissions.
\item Endpoint execution is granted or denied.
\item Attempt is logged for traceability.
\end{enumerate}

\section{Realization}

\subsection{Backend Implementation Details (Spring Boot)}

Sprint 1 backend realization implemented:

\begin{itemize}
\item JWT token generation and validation filters
\item Role-aware endpoint protection through Spring Security
\item Account and password management endpoints
\item Centralized exception handling and standardized API error payloads
\item Initial logging and auditing hooks for identity events
\end{itemize}

Representative endpoint families introduced in this sprint include:

\begin{itemize}
\item \api{authenticate}
\item \api{account}
\item \api{account/change-password}
\item \api{admin/users} (baseline administration)
\end{itemize}

\subsection{Frontend Implementation (Angular)}

The Angular client delivered:

\begin{itemize}
\item Login page with reactive validation
\item Route guards and role-sensitive navigation
\item HTTP interceptor for token propagation
\item Unauthorized and forbidden access handling

\section{Conclusion}

Sprint 1 delivered the strategic technical base of RM by implementing secure authentication, authorization controls, foundational persistence, and client-side session management. This increment reduced project risk by stabilizing cross-platform identity flows before introducing business-specific intervention logic. Sprint 2 builds directly on this foundation to implement rapport request management and dynamic form templates.
